DWH & OLAP: Professional Refresh 🚀

A senior-level knowledge base designed for a 15-minute conceptual update.

1. Architectural Foundations 📐

Design Philosophies

Modern architecture prioritizes decoupling storage from compute. This allows for elastic scaling without migrating physical data.

Kimball vs. Inmon

  • Kimball (Bottom-Up): Organizes data into Dimensional Models (Star/Snowflake). It is highly optimized for query speed and user accessibility. It remains the industry standard for BI.
  • Inmon (Top-Down): Organizes data into a centralized Normalized (3NF) repository. It acts as a robust "Single Source of Truth" (SSOT) that ensures high data integrity across all downstream marts.

The ELT Paradigm

In the cloud era, we use ELT. We load raw data directly into the warehouse and use its massive parallel processing (MPP) power to perform transformations using SQL or Python (e.g., via dbt or Spark).

2. OLAP: Multidimensional Reality 🧊

Short Reference

OLAP engines provide the logic required to analyze data across multiple dimensions (Time, Geography, Product) simultaneously.

Storage Mode Mechanism Optimal Use Case
MOLAP Pre-calculated data cubes. High-speed access to static, predictable metrics.
ROLAP Dynamic SQL against relational tables. Real-time analysis on high-volume, granular data.
HOLAP Hybrid approach. MOLAP for summaries; ROLAP for deep-dive detail.

Standard Operations

Visualizing the data cube through five core actions:

  • Roll-up: Summarizing (e.g., moving from City to Country).
  • Drill-down: Increasing detail (e.g., moving from Year to Month).
  • Slice: Selecting a single dimension (e.g., "Only Sales for 2026").
  • Dice: Selecting a sub-cube (e.g., "Shoes in Israel during Q1").
  • Pivot: Rotating the axes to change the analytical perspective.

3. Modern Trends (2026) 🌟

Strategic Shifts

The Data Lakehouse 🏠

The Lakehouse architecture merges the low cost of a Data Lake with the performance and ACID transactions (Atomicity, Consistency, Isolation, Durability) of a Data Warehouse. By using open table formats like Apache Iceberg, you avoid vendor lock-in while maintaining DWH-level speed.

Headless BI (Semantic Layer) 🧠

Instead of defining business logic (like "Active Users") inside individual BI tools, you define it once in a Semantic Layer (e.g., dbt MetricFlow or Cube). This ensures that whether you use Power BI, Tableau, or a Python notebook, the numbers are identical.

Data Mesh 🕸️

A shift from centralized data teams to Domain-Driven Ownership. Business units (like Marketing or Finance) own their data as a "Product," providing it to the rest of the company via standardized APIs or shared catalogs.

4. Advanced Engineering Patterns 🛠️

Senior Engineering

The Medallion Architecture

A data governance pattern that provides structure and lineage to the Lakehouse:

  • Bronze (Raw) 🥉: The landing zone. High-fidelity, immutable data exactly as it arrived from the source.
  • Silver (Validated) 🥈: The "Integration" layer. Data is cleaned, standardized, and joined. This is where cross-domain consistency is established.
  • Gold (Curated) 🥇: The "Reporting" layer. Data is modeled into final aggregates or Star Schemas for specific business use cases.

Slowly Changing Dimensions (SCDs)

Crucial for maintaining historical accuracy in reporting:

  • Type 1: Overwrites existing data. No history is kept.
  • Type 2: Adds a new record with start_date and end_date. The industry standard for full history tracking.
  • Type 4: Maintains the current state in one table and moves historical changes to a dedicated "History" table.

Data Contracts

A proactive agreement between data producers (source systems) and consumers (DWH). It defines the schema, metadata, and quality requirements, preventing "silent failures" when source systems change.
